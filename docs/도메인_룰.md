### 룰

- 법인 사업자 코드는 무자료 거래 발생 시 입력 자체를 막는다 (시스템에서는 validation).

- **매입 부가세 자동 분기** — `2.매입` N열(부가세) 수식이 사업자(Q) 또는 출금통장(X) 중 하나라도 `B계좌`면 부가세 0. 무자료 매입은 부가세 신고 X 일관.

## 2. 매출 상태 머신

매출은 3단계로 진행되며, 트리거 컬럼들의 OR 조건으로 상태 전이.

```
       ┌──────┐
       │ 주문 │  거래처(D) 입력만 된 상태
       └──┬───┘
          │ 납품 트리거 OR 셋 중 하나
          │  - M열 거래명세서 = "O"
          │  - P열 세금계산서 = "O"
          │  - R열 입금통장 = "B계좌" (무자료 거래의 납품 증거)
          ▼
      ┌────────┐
      │납품완료│  N열 수금예정일 ≥ 오늘
      └──┬─────┘
         │ N열 수금예정일 < 오늘                 O열 수금완료 = "O"
         ▼                                          ▼
     ┌──────┐  ──── O열 수금완료 = "O" ────► ┌────────┐
     │ 연체 │                                │수금완료│
     └──────┘                                └────────┘
```

### 핵심 통찰

- **B계좌 입금이 납품 완료 트리거 중 하나** — 무자료 거래에는 거래명세서·세금계산서가 안 나오므로, "B계좌에 돈이 들어왔다"는 사실 자체가 납품의 증거.
- **주문 단계에서는 미수금 0** — 회계상 매출 인식 X.
- **수금완료가 모든 것을 종료시킨다** — 어느 상태에서든 O=O이면 즉시 수금완료.

### 수식 (참고, 단가 추가 후 실제 셀 letter)

```
상태 = IF(AND($D="", $E=""), "",                    # 거래처와 현장 둘 다 비면 빈칸
        IF($Q="O", "수금완료",                        # 수금완료=O → 수금완료
          IF(OR($O="O", $R="O", $T="B계좌"),         # 거래명세서/세금계산서/B계좌 → 납품 완료
            IF(AND($P<>"", $P<TODAY()), "연체", "납품완료"),
            "주문")))
```

**거래처 OR 현장 트리거**: 콜드 prospecting (현장만 있고 거래처 미상)도 상태 평가됨. 거래처 정식명 확보 시 D열에 채워 정합 회복 권장.

## 3. 매입 상태 머신

매출과 대칭 구조. 컬럼 이름만 다름.

```
   발주 → 입고완료 → 결제완료
              ↓ 결제예정일 지남
           결제연체
```

| 단계 | 트리거 |
|---|---|
| 발주 | 매입처(D) 입력만 |
| 입고완료 | M=O (거래명세서수취) OR P=O (세금계산서수취) OR R=B계좌 |
| 결제완료 | O=O (결제완료) |
| 결제연체 | 입고완료 + N(결제예정일) < 오늘 |

## 5. 미수금 / 미지급금 계산

```
미수금 = IF(금액=0, 0,
          IF(수금완료=O, 0,
            IF(거래명세서=O OR 세금계산서=O OR 입금통장=B계좌,
               금액,
               0)))
```

즉, **납품 완료된 상태에서 수금이 안 됐을 때만** 미수금이 잡힌다. "주문" 단계는 미수금 0.

미지급금도 동일 패턴 (입고완료 + 결제 미완료).

**6월 시스템에서는 generated column으로 처리**:
```sql
outstanding_amount NUMERIC GENERATED ALWAYS AS (
  CASE
    WHEN amount IS NULL OR amount = 0 THEN 0
    WHEN payment_received THEN 0
    WHEN delivery_note OR tax_invoice OR account_in = 'B'
      THEN amount
    ELSE 0
  END
) STORED
```

## 8. 통장 누적잔고 계산

`6.통장.G열`(누적잔고)는 분류로 분기:

### 시작잔고 행 (분류='시작', 보통 행 2~4)
그 통장의 **전체 잔고**(시작잔고 + 모든 거래 누적). 통장별 현재 잔고 한눈 표시:

```
G2 (법인A 시작) = SUMIFS(E:E, C:C, "법인A") - SUMIFS(F:F, C:C, "법인A")
```

### 거래 행 (row 5+)
같은 통장의 row 2부터 현재 row까지 누적 (시간 순 잔고 흐름):

```
G5 = SUMIFS($E$2:E5, $C$2:C5, C5) - SUMIFS($F$2:F5, $C$2:C5, C5)
```

### 사용 패턴
- 시작잔고 행 입금 컬럼에 실제 시작 금액 입력 → 모든 잔고 계산이 자동 정확
- 거래 행은 자동 미러링(`migrate.py mirror`) 또는 수동 입력
- 시작잔고 G2/G3/G4를 보면 통장별 현재 잔고가 즉시 보임 (별도 잔고 요약 영역 L~O와 중복이지만 시각적 가독성)

**6월 시스템에서는**: `bank_transactions` row 단위 + materialized view로 잔고 계산. 또는 `bank_balance_snapshots` 일별 스냅샷. 워크북의 시작잔고 입금 amount는 시스템에서 `bank_accounts.starting_balance` 컬럼으로 직접 매핑.

### 명함 → 거래처 promote 흐름

1. 명함 수령 (10.명함에 행 추가, 상태=대기)
2. 후속 영업 활동 (9.영업내역에 활동, 활동유형=전화/방문/견적 등)
3. 거래 가능성 확인 → 상태=팔로업중
4. 첫 거래 발생 → 5.거래처에 회사 등록 + 10.명함 상태=거래시작

### 6월 시스템 매핑

```sql
CREATE TYPE contact_status AS ENUM (
  'PENDING', 'FOLLOWUP', 'PROMOTED', 'LOST', 'ON_HOLD'
);

CREATE TABLE contacts (
  id UUID PRIMARY KEY,
  code TEXT GENERATED (CARD-YYYYMMDD-NNN),
  name TEXT NOT NULL,
  company_id UUID REFERENCES parties(id),    -- nullable (콜드)
  company_text TEXT,                          -- 마스터 등록 전 자유 텍스트
  title TEXT,
  phone TEXT, email TEXT, company_phone TEXT, address TEXT,
  card_image_url TEXT,                       -- Supabase Storage
  card_image_ocr_data JSONB,                 -- OCR 파싱 결과
  received_at DATE,
  received_location TEXT,
  received_via_activity_id UUID REFERENCES sales_activities(id),
  status contact_status DEFAULT 'PENDING',
  notes TEXT
);

ALTER TABLE sales_activities
  ADD COLUMN contact_id UUID REFERENCES contacts(id);
```

### 6월 시스템 강화 기능

- **명함 사진 OCR 자동 파싱**: 모바일 사진 한 번 → 자동 데이터화
- **회사명 fuzzy matching**: 5.거래처 마스터에 비슷한 회사 매칭 제안
- **카카오톡/문자 빠른 발송**: 명함 핸드폰 → 즉시 인사
- **활동 자동 매핑**: 9.영업내역 활동유형='명함수령' + 명함 등록 시 자동 contact_id 연결
- **거래처 promote 자동화**: 첫 거래 발생 시 5.거래처 등록 + 명함 상태 PROMOTED 갱신

## 12. 영업내역 (활동 단위 로그)

`9.영업내역` 시트는 **활동 단위 로그**. 한 행 = 한 번의 영업 활동.

### 매출/매입과 다른 점

- **5.거래처 마스터 매칭 강제 X** — 콜드 prospecting (지나가다 명함 받기, 현장 방문 등) 입력을 위해 거래처/잠재처 컬럼은 자유 텍스트.
- **등록여부 자동 판별** — D열은 `IF(COUNTIF('5.거래처'!B:B, C)>0, "등록", "잠재")` 수식. 잠재처는 옅은 파랑 강조.
- **수주 시 1.매출과 연결** — 결과=수주이면 M열(매출ID)에 1.매출의 주문ID 입력 (수동, 6월 시스템에선 FK).

### 활동유형 enum

| 값 | 시스템 enum | 비고 |
|---|---|---|
| 전화 | `CALL` | |
| 방문 | `VISIT` | 약속된 방문 |
| 견적 | `QUOTE` | 견적서 발행 |
| 카톡 | `KAKAO` | |
| 문자 | `SMS` | |
| 명함전달 | `CARD_GIVEN` | 우리 명함 전달 |
| 명함수령 | `CARD_RECEIVED` | 상대 명함 수령 |
| 현장콜드 | `COLD_VISIT` | 약속 없이 현장 방문 |
| 기타 | `OTHER` | |

### 결과 enum

| 값 | 시스템 enum | 색상 |
|---|---|---|
| 진행중 | `IN_PROGRESS` | 노랑 |
| 수주 | `WON` | 녹색 |
| 실주 | `LOST` | 회색 |
| 보류 | `ON_HOLD` | 옅은 주황 |

### 다음 follow-up overdue

`AND(결과="진행중", 다음_follow_up < TODAY())` → N열 빨강 강조. 미처리 follow-up 알림.

### 6월 시스템 매핑

```

  + parties.kind ENUM에 'PROSPECT' 추가
  + 잠재처는 마이그레이션 시 parties(kind=PROSPECT)로 자동 INSERT (이름 dedup)
  + 거래 발생 시 PROSPECT → CUSTOMER로 promote
  + 영업내역.party_id는 항상 FK (PROSPECT든 CUSTOMER든 parties에 존재)
```

워크북의 `등록여부` 컬럼은 시스템에선 `parties.kind`로 흡수 (제거).

## 13. 한국 세무 컨텍스트 (요약)

### 부가세 신고
- 분기별 신고 (4/25, 7/25, 10/25, 1/25)
- 일반과세자: 매출 - 매입의 부가세 차액 납부
- B계좌 거래는 매출에서 제외 → **친구 개인사업자에서만 발생, 법인 절대 X**

### 의제매입세액공제
- 면세 농수산물 매입 시 부가세 공제 가능
- 철강 유통에는 거의 무관 — 단, 폴리싱 작업 시 면세 자재 사용 가능성 검토 필요

### 특수관계자 거래
- 법인과 친구 사업자 사이 거래는 시세 기반 필수
- 부당행위 판정 시 추징 위험
- 한 거래처 비중 70% 이하 유지 룰

### 가공 세금계산서 절대 금지
- 실거래 없는 세금계산서 발행 → 조세범처벌법 (3년 이하 징역)
- 친구가 과거에 했더라도 동업 시작 후에는 0%
