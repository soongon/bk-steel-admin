# 배포 가이드 — Vercel 프로덕션

> 결론: **기존 신라철강 홈페이지/블로그 프로젝트에 통합하지 말고, 별도 Vercel 프로젝트 + `admin.도메인` 서브도메인.**
> 이 앱은 공개 사이트가 아니라 Supabase Auth로 막힌 내부 운영 어드민이고, 매출·매입·재고에 더해 **B계좌 무자료 흐름**까지 다룬다. 공개 마케팅 배포와 빌드·env·인증·롤백 단위를 섞을 이유가 없다. Vercel은 같은 팀 아래 N개 프로젝트가 무료이므로, "별도 프로젝트"가 곧 실질적 통합 관리다.

## 1. 배포 경로 — GitHub 연동 (권장)

origin: `https://github.com/soongon/bk-steel-admin.git`

1. Vercel 대시보드 → **Add New… → Project**
2. **Import Git Repository** → `soongon/bk-steel-admin` 선택
3. **Framework Preset**: Next.js (자동 감지) · **Root Directory**: `./` · **Build Command**: `next build` (자동)
4. 아래 [환경변수](#2-환경변수) 입력 후 **Deploy**
5. 이후 `git push origin main` → **자동 프로덕션 배포**, PR → Preview 배포

> `vercel.json`·`next.config` 커스텀 불필요(zero-config). 이미 로컬 `npm run build` 통과 확인됨.

## 2. 환경변수 (Vercel → Settings → Environment Variables)

| 키 | 값 출처 | 환경 | 비고 |
|---|---|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` | `.env.development` | Production · Preview · Development | 공개 키 |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | `.env.development` | 동일 | 공개(브라우저 노출 전제, RLS로 보호) |
| `SUPABASE_SERVICE_ROLE_KEY` | `.env.local` | (선택) Production | **레이더/시드를 서버에서 돌릴 때만.** ⚠️ Sensitive 체크 |
| `DATA_GO_KR_BUILDING_KEY` | `.env.local` | (선택) | 발주 레이더 건축HUB |
| `DATA_GO_KR_NARA_KEY` | (미설정 시 building 키 fallback) | (선택) | 발주 레이더 나라장터 |
| `SOLAPI_API_KEY` · `SOLAPI_API_SECRET` · `SOLAPI_SENDER` | `.env.local` | (선택) Production | 명세서 MMS 전송. ⚠️ KEY/SECRET Sensitive |
| `POPBILL_LINK_ID` · `POPBILL_SECRET_KEY` | `.env.local` | (선택) Production | 전자세금계산서 발행. ⚠️ Sensitive. 빈값이면 mock |
| `POPBILL_IS_TEST` | `true`(기본) | (선택) Production | `true`=테스트베드(국세청 실전송 X). **운영전환 승인 후에만 `false`** + redeploy |
| `POPBILL_SENDER` | 팝빌 등록 발신번호 | (선택) Production | 문자 MMS·알림톡 발신. 미설정 시 명세서 MMS 는 솔라피 fallback |
| `POPBILL_ATS_TPL_INVOICE` · `POPBILL_ATS_TPL_DUNNING` | 팝빌 승인 템플릿코드 | (선택) Production | 알림톡(발행안내/입금안내). 미설정 시 해당 알림톡 미발송 |

> ⚠️ **가장 흔한 실수**: `.env.development`는 `next build`(프로덕션)에서 **로드되지 않는다**(`next dev` 전용). 따라서 `NEXT_PUBLIC_*` 두 개를 반드시 Vercel에 직접 등록해야 한다. 누락 시 빌드는 통과해도 런타임에 Supabase 연결이 끊긴다.
>
> `SUPABASE_SERVICE_ROLE_KEY`는 app/·components/·lib/ 어디서도 import하지 않는다(마이그레이션·레이더 스크립트 전용). 앱 런타임에는 불필요하므로, 레이더를 Vercel cron/route로 돌릴 계획이 없으면 등록하지 않아도 된다.

## 3. Supabase

현재 dev/prod가 **단일 인스턴스**(`qjbdtxwvgpldvdmmorcp`)다. 둘 중 하나를 택한다:

- **옵션 A — 이대로 프로덕션 사용**: 빠르고 단순(소규모/초기 운영). 개발 중 실수 변경이 운영 데이터에 직접 닿는 위험.
- **옵션 B — 새 prod 프로젝트 분리** (권장): 운영/개발 데이터 격리. 단 마이그레이션(`supabase db push`)과 시드(품목·rebar_spec 등) 재실행 필요.

공통 후처리:
- Supabase → **Authentication → URL Configuration**: Site URL/Redirect URLs에 `https://admin.도메인` 추가
- RLS 정책(`0019_rls.sql`·`0041`·`0043`)이 대상 DB에 적용됐는지 확인(단일 인스턴스면 이미 적용됨)
- 직원 계정만 Supabase Auth(이메일/비번)로 발급. 가입 공개 금지

## 4. 도메인

- Vercel → 프로젝트 → **Settings → Domains** → `admin.기존도메인` 추가
- DNS(기존 도메인 관리): `CNAME  admin → cname.vercel-dns.com`
- 공개 홈페이지 프로젝트와 도메인/배포 완전 분리 유지

## 5. 배포 후 확인

- `https://admin.도메인/login` 로그인 → `/all/dashboard` 진입
- 각 책(법인 `bk` / 사업자 `sl` / B계좌 `b`) 데이터 표시
- 매출·매입 생성 → 라이프사이클 패널·진행률 배지 동작
- 비로그인으로 `/all/...` 접근 시 `/login` 리다이렉트(proxy.ts)

## 6. 알려진 부채

- `npm run lint` 실패(React 19 `react-hooks/set-state-in-effect` 등 다수 폼) — **`next build`는 통과**하므로 배포엔 무관. 추후 정리.

## 부록 — CLI 배포(대안)

```bash
npx vercel login          # 브라우저 OAuth (대화형 — 직접 실행)
npx vercel link           # 프로젝트 생성/연결
npx vercel env add NEXT_PUBLIC_SUPABASE_URL production   # 키별 반복
npx vercel --prod         # 프로덕션 배포
```
