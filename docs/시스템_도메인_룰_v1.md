## 1. 3축 분리 — 책(book)

시스템 최상위 차원은 **책(book)**. 모든 거래성 row는 정확히 한 책에 속한다.

| 내부 코드 | 표시명 | 실체 | 자료성 |
|---|---|---|---|
| `bk` | **법인** | 새로 설립할 SH철강(가칭) 법인 (50:50 동업) | 100% 자료거래 강제 |
| `sl` | **사업자** | 친구 개인사업자 SL철강 | 자료거래 원칙, 일부 무자료 허용 |
| `b` | **B계좌** | 친구 사업자의 무자료 거래 흐름 (히든 통장 기반) | 무자료만 |

### 1.1 코드 vs 표시 분리

- **DB·API·URL** : `bk` / `sl` / `b` 만 사용
- **UI 출력** : 라벨 컴포넌트 하나로 통일 (`<BookBadge book="bk" />` 등)
- 라벨/색상은 `lib/book.ts` 한 파일에서 중앙 관리

```ts
export const BOOK_LABEL = { bk: '법인', sl: '사업자', b: 'B계좌' } as const
export const BOOK_COLOR = { bk: 'blue', sl: 'amber', b: 'red' } as const
```

### 1.2 책 컨텍스트는 라우트에 박는다

```
/bk/sales         /bk/purchases     /bk/inventory   ...
/sl/sales         /sl/purchases     /sl/inventory   ...
/b/cash           /b/inventory      ...
```

### 1.3 마스터 데이터 책임 정리

| 마스터 | 결정 |
|---|---|
| 거래처 | **공유** (book별 신용한도/메모는 분리) |
| 품목/규격 | **공유** |
| 통장 | **책 종속** (법인 통장 ≠ 사업자 통장 ≠ B통장) |
| **창고** | **공유 + zone으로 책 구분** ← 결정 |

#### 창고 구조

```sql
warehouse                       -- 공유 마스터
  id, name, address
  kind enum('owned','partner')  -- 자사 야적장 / 타사 야적장(위탁)
  partner_id uuid               -- kind='partner'일 때 거래처 참조

-- 모든 재고 라인은 warehouse + zone + book으로 위치/소유 표현
purchase_line.warehouse_id      -- 어느 창고
purchase_line.zone              -- 책별 구획 (예: 'Zone-A')
purchase_line.book              -- 소유 책 (bk/sl/b)
```

## 2. 책 간 이관 매트릭스

| From → To | **BK** | **SL** | **B** |
|---|:-:|:-:|:-:|
| **BK →** | — | ✅ 정상거래(`inter_book_transfer`) | ❌ 금지 |
| **SL →** | ✅ 정상거래(`inter_book_transfer`) | — | ✅ 비공식이관(`internal_reclass`) |
| **B →**  | ❌ 금지 | ✅ 비공식이관(`internal_reclass`) | — |

### 2.1 거래 타입

| 타입 | 사용 사례 | 자료 발행 | 부가세 영향 |
|---|---|:-:|:-:|
| `sale` / `purchase` | 외부 거래 | book별 정책 (§3·§4) | book별 (B만 제외) |
| `inter_book_transfer` | BK ↔ SL | ✅ 필수 (특수관계자 거래) | ✅ |
| `internal_reclass` | SL ↔ B | ❌ | ❌ |
| `inventory_adjustment` | 절단·가공·재고실사 차이 | — | — |

### 2.2 확정된 핵심 룰

- **BK ↔ B 차단** : 법인이 무자료 거래에 노출되면 안 됨. DB constraint로 강제.
- **BK ↔ SL `inter_book_transfer`** : 특수관계자 거래라 **시가 산정 근거 첨부 필수**(강제). 시세 이력 ID 또는 외부 견적서 첨부 없으면 거래 생성 불가.
- **SL ↔ B `internal_reclass`** : 같은 사업주(친구)의 자산 재분류라서 매출/매입 통계에 잡히지 않음. **가격은 시세 이력 기반 시가 자동 적용**(없으면 매입원가 fallback). audit_log + `transfer_pair_id`로 이력 보존.

## 3. 책별 거래 정책 (자료성)

거래성 row 공통 컬럼 `is_documented bool` + `tax_doc_type` (§4) 로 강도 구분.

| 책 | 자료 비율 원칙 | 무자료 row 허용? | 부가세 신고 |
|---|---|:-:|:-:|
| **BK** | 100% | ❌ (DB constraint로 차단) | 포함 |
| **SL** | 원칙 자료, 일부 무자료 허용 | ✅ (row 단위 `is_documented=false` 플래그) | `is_documented=true` row만 포함 |
| **B** | 무자료만 | 모든 row 무자료 | 제외 |

**SL의 무자료 row**는 B 책으로 이동시키지 않고 **SL 책 내에 플래그로 표시**. SL과 B는 별개 사이클(폐쇄)로 유지.

## 5. 재고 도메인

### 5.1 모델 — 단일 ledger

별도 piece 테이블은 두지 **않는다**. 매입라인이 곧 가닥/lot 역할, 매출라인 ↔ 매입라인 allocation으로 추적.

```
purchase                                  -- 매입 헤더
purchase_line                             -- 매입 라인 = 입고 단위 (piece 역할)
  id, purchase_id, book, warehouse_id, zone, item_id
  acquired_unit  enum('ton','kg','ea','piece')
  acquired_qty   numeric
  unit_price     numeric
  length_mm, grade, ...                   -- 가닥 단위면 채움
  theoretical_weight_kg                   -- 이론중량
  actual_weight_kg                        -- 실중량 (저울 계량, nullable)
  price_basis    enum('theoretical','actual')
  invoiced_weight_kg                      -- 세금계산서·거래명세표 발행 기준
  status         enum('ordered','in_stock','partial_out','depleted','transferred_out','scrapped')

sale_line
  id, sale_id, book, item_id, qty, unit, weight_kg, ...
  theoretical_weight_kg, actual_weight_kg, price_basis, invoiced_weight_kg

sale_line_allocation                      -- 어느 매입라인에서 얼마나 차감했는지
  id, sale_line_id, purchase_line_id, allocated_qty, allocated_weight_kg

inventory_adjustment                      -- 절단·가공·실사 차이·반품 등
  id, book, type, ...
  type enum('cut','split','stocktake','loss','scrap','return_in','return_out')
```

### 5.2 재고 = `purchase_line` − `sale_line_allocation`

뷰/MV로 노출. 책별·창고별·zone별·품목별 집계.

### 5.3 절단/가공

`inventory_adjustment(type='cut', parent_purchase_line_id=..., children=[{length, weight}, ...])`. 원본 보존, 자식 라인이 새 재고가 됨.

### 5.4 평가법

**개별법(specific identification)**. `purchase_line` ↔ `sale_line_allocation` 으로 정확히 매칭되므로 자연스럽게 개별법 성립.

### 5.5 위탁보관 / 타사 야적 (v1 포함)

두 방향 모두 지원:

| 케이스 | 모델링 |
|---|---|
| **우리 재고가 타사 야적에 있음** (consigned_out) | `purchase_line.warehouse_id` 가 `kind='partner'` 인 창고를 가리킴. 소유는 우리 book, 위치만 외부 |
| **타사 재고가 우리 야적에 있음** (consigned_in) | 별도 테이블 `consignment_in` (우리 ledger 외부, 책 무관). 보관 책임만 추적 |

```sql
consignment_in                            -- 우리가 보관 중인 타사 재고
  id, partner_id (거래처), warehouse_id (우리 창고), zone
  item_id, qty, unit, weight_kg
  in_at, out_at  -- 입고/출고 시각
  status enum('in','out')
  notes
```

이 테이블은 결산·신고와 무관하지만 야적장 공간 관리 + 사고 시 책임 소재에 필요.

### 5.6 선급·예약·발주중 (v1 포함)

매입 상태머신: `ordered → in_stock → partial_out → depleted`
매출 상태머신: `reserved → confirmed → delivered → settled`

- `purchase_line.status = 'ordered'` : 발주 완료, 미입고 → 재고에 잡히지 않지만 "예정 입고"로 노출
- `sale_line.status = 'reserved'` : 매출 예약, 미출고 → 재고에서 "예약 차감" (allocation 생성, 단 status가 reserved)

## 6. 중량 기준 (이론중량 vs 실중량)

### 6.1 한국 철강 유통 현실

- **이론중량(theoretical)** : 규격 × 길이 × 표준비중 계산값
- **실중량(actual)** : 야적장 저울 실측값
- 압연공차(±3~5%), 길이공차, 표면 산화·페인트로 둘은 항상 다름

### 6.2 거래 방향·단위별 중량 기준 (자동 분기)

| 거래 방향 | 단위 | 시나리오 | 기본 `price_basis` | `actual_weight_kg` |
|---|---|---|:-:|:-:|
| **매입** | 톤 | 제강사 정품, 묶음 | `theoretical` | 선택 |
| **매입** | kg | 중고철근 등 | `actual` | **필수** |
| **매입** | 가닥/EA | H빔·각파이프 등 | `theoretical` | 선택 |
| **매출** | 톤/EA/가닥 | 거의 모든 매출 | `theoretical` | 선택 |

사용자가 거래마다 override 가능. 차이(`weight_diff = actual − theoretical`)는 자동 계산되어 거래처별 평균 차이를 메트릭으로 노출.

### 6.3 재고 자산가치 평가

- 실중량(`actual_weight_kg`)이 있으면 그 값 기준
- 없으면 이론중량 기준
- 평가액 = 잔량 × 그날의 시세 (§7) — **책별로 노출**

## 7. 시세(price_history) 정책

**혼합 모델** — 수동 입력 + 매입가 자동 누적 + (v2) 외부 시세 import.

### 7.1 데이터 모델

```sql
market_item                               -- 시장 표준 분류 (시세 단위)
  id, code, label_ko, category, default_unit

item.market_item_id                       -- 우리 품목이 어느 market_item에 속하는지

price_history
  id, market_item_id, recorded_on (date)
  price_per_unit, unit ('kg'|'ton')
  price_type enum('wholesale','retail','scrap','spot')
  source enum('manual','external','purchase_derived')
  source_label, source_url
  recorded_by, notes
  unique (market_item_id, recorded_on, source, price_type)
```

**`market_item` 분류는 본인 정의 실무 분류** (예: `철근 D13`, `H빔 200×100`, `각파이프 50×50×2.3`, `고철 생철`). 시장 표준 분류는 따르지 않음.

### 7.2 데이터가 채워지는 3가지 경로

| 경로 | 방식 | source | v1? |
|---|---|---|:-:|
| A. 수동 입력 | "오늘의 시세" 페이지에서 사용자가 직접 | `manual` | ✅ |
| B. 매입가 자동 누적 | 매입 등록 시 그 단가가 동일 날짜로 자동 row 추가 | `purchase_derived` | ✅ (ON) |
| C. 외부 시세 import | CSV/paste 업로드 (스틸데일리 등) | `external` | ❌ (v2) |

### 7.3 "오늘의 시가" 선정 우선순위

`manual` > `external` > `purchase_derived`. 같은 날 여러 source 공존 가능.

### 7.4 입력 UX — "오늘의 시세" 페이지 (v1 포함)

- `/admin/prices/today` (책 무관, 공유 페이지)
- 즐겨찾는 8~12개 품목군(큐레이션) 한 화면 노출
- "어제와 동일", "일괄 +%" 버튼 등 빠른 입력
- 미입력 날은 가장 최근값을 carry-over하여 대시보드 표시
- 큐레이션 목록은 별도 화면에서 사용자가 직접 관리

## 8. 대시보드 KPI

- 재고 회전율
- **거래처 신용한도 대비 미수 잔액** (한도 초과 알림)

## 10. v2 이연 기능

다음 기능은 v1에 포함하지 않고 v2(또는 그 이후)로 이연한다. v1에서 발생하면 수동 조정 거래(`inventory_adjustment`, credit/cancel row 등)로 처리한다.

| 기능 | 이연 사유 |
|---|---|
| **반품·클레임 워크플로우** | v1에선 수동 조정으로 처리. 빈도 보고 v2에 정식 워크플로우 |
| **할인·리베이트** | v1은 거래 단가에 직접 반영. 사후 정산 리베이트는 v2 |
| **품질 등급 재분류** | 빈도 매우 낮음. 발생 시 수동 조정 |
| **시스템 내 분개 더블엔트리** | 회계 시스템과 충돌 가능성. export만으로 시작 |
| **외부 시세 자동 import** | 크롤링·약관 이슈. 우선 수동/매입가 기반 운영 |
