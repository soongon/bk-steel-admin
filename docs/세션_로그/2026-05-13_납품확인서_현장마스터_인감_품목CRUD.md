## 1. B계좌는 SL 사업자의 무자료 흐름 (alias 확정)

### 결정
- BK · SL 의 외부 문서(거래명세표 · 세금계산서 · 향후 견적서)는 각 책의 `company_profile` 사용.
- **B계좌 거래는 항상 SL 정보로 발급**. B계좌는 SL 사업자가 운영하는 무자료 흐름이라 사업자 인격은 SL과 동일.

### 구현
- `lib/company-profile.ts` 의 `fetchCompanyProfile(book)` 에 `effectiveBook(book)` 매핑 도입 — `book === 'b'` 면 `sl` row 반환.
- 회사 정보 페이지의 B 카드는 SL row 데이터를 read-only 로 표시, 편집 버튼 제거, "사업자 정보 사용" 배지 + 안내 박스.
- DB의 b row 는 그대로 두되 어느 경로에서도 사용 안 됨 (cleanup 검토는 Phase 3 이후).

### 함정 (commit `42242d1`)
- 동일 메시지 내 `Edit` 2건 중 첫 번째가 "File has not been read yet" 으로 **무음 실패** 한 채 두 번째만 적용되어 푸시됨 → 매핑이 실제로 적용 안 된 상태였음.
- 사용자 보고로 재현 시 `git log` 의 diff 와 실제 파일을 비교해 누락 확인 후 `42242d1` 으로 재적용.
- **교훈**: 같은 메시지에서 여러 파일을 동시에 Edit 한 경우, 일부 실패가 다른 성공 결과에 가려질 수 있음. 결과 메시지에서 명시적으로 모든 파일의 성공을 확인할 것.

## 2. 납품확인서 (Delivery Certificate) 도메인 + 1단계 골격

### 도메인 룰 (commit `8133b35`)

- 철근이 단 1원이라도 납품되면 **의무 발급**. 자료/무자료 무관.
- **거래처 + 현장 단위로 1회** 발급.
- 발급 시점은 납품 시가 아니라 거래처 요청 시 (보통 준공검사 직전).
- 준공검사 첨부 필수 → 거래처가 납품대금을 떼이지 않는 **구조적 안전장치**.
- 사업자 인감 첨부 필수.

### DB (`0030_delivery_certificate.sql`)
- `delivery_certificate (id, book, partner_id, site_name(legacy), doc_no, issued_on, issued_by, notes, soft-delete, audit, updated_at)`
- 초기 UNIQUE: `(book, partner_id, COALESCE(site_name, ''))` — 1회 발급 룰 강제. (현장 마스터 도입 후 `site_id` 기준으로 변경됨 — §3 참조)
- `sale.delivery_cert_id` FK ON DELETE SET NULL — 어느 cert 에 묶였는지.
- RLS: viewer 읽기 / staff 발급·수정 / manager 삭제.

### 발급 로직
- 매출 상세 페이지 `[납품확인서 발급]` 클릭 → server action.
- 동일 (book, partner, site) 의 모든 미발급 sale 에 cert_id 일괄 부여.
- 이미 cert 존재하면 그 cert 재사용 (재발급 모달은 동일 row 재표시).

### UI (1단계)
- 매출 목록: doc_no 셀 아래 `✓ 확인서` 배지 (cert_id NOT NULL).
- 매출 상세 액션바: `[발급]` / `[보기]` 토글 + 모달 (placeholder 양식, 다음 단계에서 본체).

## 3. 현장(Site) 마스터 격상 — Phase 1 + 2

### 사용자 결정
- **독립 마스터 (partner와 M:N)**. 한 현장에 시공사·하청·자재상 등 여러 거래처 등장 가능.
- 따라서 site 는 글로벌 (book 컬럼 없음) — partner 와 동일 패턴.
- 단계: **Phase 1만** (마스터·사이드바·CRUD·매출 폼 콤보).

### DB (`0031_site.sql`)
- `site (id, code 'S-NNNN' auto, name UNIQUE, address, city, client_name, status, started_on, ended_on, notes, is_active, audit, soft-delete)`
- 코드 자동: `site_code_seq` → `S-0001` 패턴 (partner 동일).
- partner 와 동일 RLS — 인증 사용자 SELECT, owner/manager 만 변경.
- `sale.site_id`, `delivery_certificate.site_id` FK 추가.
- **데이터 마이그레이션**: 기존 `sale.site_name` distinct → `site` 자동 시드, sale·cert 의 site_id 자동 채움. UNIQUE 키도 `site_name` → `site_id` 기준으로 갱신.

### UI (Phase 1)
- `/[book]/sites` CRUD 페이지 — partner 패턴 그대로.
- 사이드바 "공유 마스터" 그룹에 **현장** 메뉴 (MapPinIcon).
- 매출 폼 "현장" input: datalist 자동완성 + 미등록 입력 시 "저장 시 자동 생성" 안내.
- 매출 actions: `resolveSiteId(siteId, siteName)` — id 가 없으면 name 으로 site 자동 생성 후 id 부여.
- 매출 목록: `site.name + S-NNNN` 코드 표시 (legacy `site_name` fallback).

### Phase 2 (commit `fa03163`)
- `/[book]/sites/[id]` 현장 상세 페이지 — 메타 + 거래처×책 그룹별 카드 (매출 라인 라이트 표시 + 합계 + cert 상태) + [거래명세표] [납품확인서] 버튼.
- `/[book]/sites/[id]/statement?partner=&book=` — 거래처별 누적 거래명세표 (A4 인쇄용). 여러 매출 라인을 1장에 누적, 라인별 월/일 표시.
- `/[book]/sites/[id]/cert-actions.ts` — `issueDeliveryCertBySite(book, partner, site)` server action.
- `components/admin/delivery-cert-form.tsx` — 표준 납품확인서 양식 본체. 수신(거래처) + 현장 박스 + 본문(납품기간) + 납품내역 표(라인별 매출번호 포함) + 공급자+직인.
- `TradingStatement` 의 `StatementLine` 에 `ordered_on` 추가 → 누적 명세표에서 라인별 날짜 표시.
- 현장 목록 → 행에서 현장명/코드 클릭 → 상세로 진입 (commit `7943b6c` — 첫 빌드 시 link 누락 버그 수정).

### legacy 정리 (Phase 3 후보)
- `sale.site_name`, `delivery_certificate.site_name` 컬럼은 표시 fallback 용도로 당분간 유지.
- 새 매출에서는 `site_id` 정답, `site_name` 도 함께 채워 동기화. 사용 안 정착 후 컬럼 제거 검토.

## 8. 품목 CRUD 단건 (commit `0f768dc`)

- 기존 PageStub 였던 `/items` 를 partner 패턴의 CRUD 로 구현. DB 변경 없음 (item 테이블은 0004 에 이미 정의됨).

### 폼 (카테고리별 분기)
- 카테고리: rebar / hbeam / pipe / scrap / etc 드롭다운으로 입력 필드 동적 분기.
- **rebar**: spec_code(드롭다운 — rebar_spec 시드 unit_weight 표기) + grade_code(드롭다운) + length_m → **코드·이름 자동 생성** (`REBAR_D13_SD400_8M` / `철근 D13 SD400 8M`). 미리보기 박스에 가닥당 중량 환산도 즉시 표시.
- **그 외**: 코드·이름 직접 입력 + spec_text 자유.
- 모두: `weight_per_unit_kg`(선택), `is_active`.

### 목록
- 카테고리 컬러 뱃지 (철근/H빔/각파이프/고철/기타).
- 카테고리 필터 드롭다운.
- 규격 통합 표시 (rebar 는 spec + grade + length / 그 외는 spec_text).
- 단위중량, 활성 상태.

### 미완료 (내일)
- **품목 일괄 인서트** — rebar 의 (spec × grade × length) 매트릭스가 많아 단건 등록이 비효율. 사용자가 내일 진행하기로 함. CSV 업로드 / SQL 시드 / 매트릭스 UI 중 어느 쪽으로 갈지 결정 후 시작.
