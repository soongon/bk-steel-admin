# 발주 레이더 (Construction Order Radar)

경주·포항·울산 권역에서 **곧 철근·강관·각파이프가 필요해질 건설 현장**을 공공데이터로 매일
자동 수집·점수화해서, 영업이 "어디에 / 언제 / 누구한테" 전화할지 한 화면에서 보는
**영업 인텔리전스 대시보드**.

운영(매출·매입·통장)과는 **별개**다. 외부 공공데이터 기반이고 **100% 정상 데이터**라
법인 쪽에 깨끗하게 붙는다. B계좌/무자료와는 전혀 무관하며 `entities`/`book` 차원과 독립이다.

> 1차 범위 = **"보는 것"까지** (수집·점수화·대시보드 표시). 리드 파이프라인(관심→접촉→견적→수주)
> 전환은 2차 보류 — 데이터 모델에 `linked_partner_id` 자리만 비워둠.

## 아키텍처 결정 (잠금)

- **독립 라우트 `/radar`** — `app/[book]/` **밖**. book(법인/사업자/B계좌)과 무관한 외부데이터라
  책 사이드바를 쓰지 않고 자체 레이아웃. (`app/login`과 같은 최상위 라우트 패턴, 인증은 `proxy.ts`가 전역 가드.)
- **수집은 tsx 스크립트** (`scripts/radar-collect.ts`). cron 트리거(Vercel/GitHub Action/서버 crontab)는 배포 시 결정.
- **지도는 리스트 먼저** — 핵심 가치(연락주체 카드 + 점수)는 지도 없이 동작. 지오코딩·지도는 phase 1.5.
- **테이블 단수명** `construction_project` (repo 컨벤션). `linked_partner_id → partner(id)` (repo엔 `customers` 없음).
- **어댑터 패턴** — `lib/radar/collectors/`. 새 소스는 어댑터 파일 하나 추가 + `COLLECTORS` 배열 등록.

## 데이터 모델 — `construction_project` (0038)

`0031_site.sql` 패턴 (gen_random_uuid PK · `set_updated_at` 트리거 · `deleted_at` soft delete · `NOTIFY pgrst`).
audit 트리거는 미부착(매일 기계 ingest라 audit_log 오염 방지).

핵심 컬럼: `source`(building_permit|nara_bid) · `source_key`(자연키) · **`(source, source_key)` UNIQUE**(upsert) ·
`region` · `project_type`(private|public) · `title`/`address`/`lat`/`lng`/`distance_km` ·
`usage`/`structure`/`floor_area`/`est_rebar_ton` · `stage` · `stage_date` ·
`ordering_org`(발주처·표시용) · **`contact_party`(연락 주체)** · `awarded_company`(낙찰사) ·
`relevance_grade`(A/B/C) · `relevance_score` · `est_amount` · `linked_partner_id`(2차용,nullable) · `raw`(jsonb).

- **RLS**: 인증 사용자 SELECT / owner·manager WRITE (site 패턴). 수집기는 service_role로 RLS 우회 upsert.
- **인덱스**: `(region, relevance_grade)` · `(stage, stage_date)`.

### 단계(stage) & 긴급도 — `lib/radar/types.ts`

| project_type | stage | 의미 | 색(긴급도) |
|---|---|---|---|
| 민간 | `permit` | 건축허가(착공 전) | 회색 — 모니터링 |
| 민간 | `construction_start` | 착공신고 | **빨강 — 지금 전화** |
| 민간 | `completed` | 사용승인(준공) | 숨김(기회 끝) |
| 관급 | `bid_notice` | 입찰공고(낙찰 전) | 회색 — 대기 |
| 관급 | `awarded` | 낙찰 확정 | **파랑 — 낙찰사에 전화** |

> **연락 주체가 핵심**: 관급은 발주처(시청)가 아니라 **낙찰사**가 철강 구매자. 카드 하단에 낙찰사를
> 파란 띠로 강조하고 "발주처 — 연락 대상 아님"을 명시해, 영업이 시청에 전화하는 실수를 구조적으로 막는다.

## 소스 어댑터 (`lib/radar/collectors/`)

### 민간 — 건축HUB 건축인허가 기본개요 (✅ 실호출 확정 2026-06)

- 엔드포인트: `https://apis.data.go.kr/1613000/ArchPmsHubService/getApBasisOulnInfo`
- 응답 경로: **`response.body.items.item[]`**, `body.totalCount`로 페이징.
- **제약 (중요)**:
  - **`bjdongCd`(법정동 5자리) 필수** — `sigunguCd`만 넘기면 빈 응답. → 시군구당 법정동 전체를 순회.
  - **날짜 파라미터·정렬 없음** + `numOfRows` 상한 **100**. → 법정동 전체를 페이징한 뒤
    클라이언트에서 **active(준공 제외) + recent(`activeWindowDays`, 기본 730일)** 만 남겨 upsert 볼륨 제어.
- **필드 매핑** (실응답 확인):

  | 우리 | API 필드 |
  |---|---|
  | source_key | `mgmPmsrgstPk` (허가관리대장 PK, 숫자) |
  | title / address | `bldNm` / `platPlc` |
  | usage | `mainPurpsCdNm` → 카테고리(공장/창고/근린/다세대/아파트/교육) |
  | floor_area | `totArea` ?? `vlRatEstmTotArea` |
  | stage | `useAprDay`(준공) → `realStcnsDay`(착공) → `archPmsDay`(허가) 순 판정 |
  | 건축구분 | `archGbCdNm` — **신축·증축만** 수집 |

  ⚠️ **주구조(`strctCdNm`)는 기본개요에 없음** → `structure`는 null. (RC/철골 판별은 동별개요 = 2차.)
- **법정동코드**: `lib/radar/bjdong-codes.ts` (시군구→bjdongCd 651개: 경주198·포항108+143·울산18+19+8+27+130).
  공식 전체자료에서 `scripts/radar-gen-bjdong.ts`로 생성(API 쿼터 0). `scripts/radar-discover-bjdong.ts`는
  "실제 인허가가 있는 법정동" 진단용(보조).

### 관급 — 나라장터 입찰공고(공사) + 낙찰 (✅ 실호출 확정 2026-06)

- 엔드포인트(**네임스페이스 다름 주의**):
  - 입찰공고: `/1230000/ad/BidPublicInfoService/getBidPblancListInfoCnstwk`
  - 낙찰: `/1230000/as/ScsbidInfoService/getScsbidListSttusCnstwk`
- 파라미터: `inqryDiv=1` + `inqryBgnDt`/`inqryEndDt`(YYYYMMDDHHMM, **범위 ≤1개월** → ≤28일 청크) · `type=json`.
  건축HUB와 달리 **날짜 필터 있음 = 준실시간**. 응답 `response.body.items[]`(배열).
- **지역 요청필터 없음** → 전국 받아 `cnstrtsiteRgnNm`(현장지역)·`dminsttNm`(수요기관)으로 거름.
- **입찰↔낙찰 조인**: 공고번호 `bidNtceNo`+`bidNtceOrd`. 낙찰 있으면 stage=`awarded`(우선), 없으면 `bid_notice`.
- **낙찰사(연락 주체)**: `bidwinnrNm`·`bidwinnrCeoNm`(대표)·**`bidwinnrTelNo`(전화)**·`sucsfbidAmt`(낙찰금액)
  → `contact_party="낙찰사 · 전화"`. **"지금 낙찰사에 전화"가 바로 됨.**
- **철근관련성**(`naraSteelCategory`): 건축 신축/구조토목만 A/B. 전기·통신·조경·포장·철거·해체·설계·MEP(설비) 등
  철근 무관 공종은 **수집 단계에서 컷**. 키워드는 `naraBid.ts`에서 보정.
- 키: `process.env.DATA_GO_KR_NARA_KEY ?? DATA_GO_KR_BUILDING_KEY` (동일 data.go.kr 계정).

## 점수 (`lib/radar/scoring.ts` + `config.ts`)

가중합(0~100) → A/B/C. **계수는 전부 `config.ts` 상수로 분리**(실거래 보정 단일 지점, 전부 초안):

- 가중치: 용도 35 · 구조 20 · 연면적 25 · 거리 20 (`SCORE_WEIGHTS`)
- 컷: A≥65, B≥45 (`GRADE_THRESHOLDS`)
- 용도 점수: 공장·창고 가산 / 대단지 아파트 감산 (`USAGE_SCORE`)
- 추정 철근톤: 연면적 × 용도별 kg/㎡ (`REBAR_KG_PER_SQM`, 철골은 `STEEL_KG_PER_SQM`). UI엔 "약 ○톤".
- 거리(`distance_km`)는 지오코딩 전이라 null → 중립(0.5) 처리. phase 1.5에서 차고지 기준 재점수.
- **관급**(`computeNaraRelevance`): 면적·구조·거리 없어 별도 — 공종 카테고리(`NARA_CATEGORY_SCORE` 건축1.0/구조토목0.6/일반토목0.2)×60 + 낙찰금액 규모(`NARA_AMOUNT_TIERS`)×40.

정렬: 긴급도(착공·낙찰) → 등급 → 점수. A등급·"지금 전화"가 위로.

**KPI**(`radar-dashboard.tsx`): 관급 낙찰(7일) · A등급 기회 · 진행 착공 현장(민간) · 추정 철근 합계.
("이번주 착공"은 건축HUB 발행지연으로 무의미 → 폐기.)

## 운영 / 스크립트

```bash
npm run radar:collect          # 전체 수집·upsert (실 DB)
npm run radar:collect:dry      # 수집·점수만, DB 미반영(요약 출력)
npm run radar:check            # 점수 순수함수 테스트
npx tsx scripts/radar-gen-bjdong.ts        # 법정동코드 재생성 → bjdong-codes.ts
```

throttle/부분 sync 환경변수: `RADAR_REGIONS`(예: `gyeongju`) · `RADAR_MAX_PAGES` · `RADAR_MAX_BJDONG` ·
`RADAR_ACTIVE_DAYS`.

**수집 비용**: 날짜 필터가 없어 법정동 전체를 페이징한다(경주 1권역 ≈ 360콜). 개발계정 쿼터를 측정한 뒤
주기를 정할 것(권장: 주 1회 풀싱크). cron 트리거는 배포 시점에 결정.

### 환경변수 (`.env.local`, gitignore)

- `DATA_GO_KR_BUILDING_KEY` — 건축HUB 건축인허가 (✅ 발급됨). **반드시 Decoding 키** (코드가 1회 인코딩).
- `DATA_GO_KR_NARA_KEY` — 나라장터 입찰+낙찰 (⏳ 대기).
- `SUPABASE_URL` / `SUPABASE_SERVICE_ROLE_KEY` — 수집기 upsert(RLS 우회).

## 남은 일

1. **관급 키** `DATA_GO_KR_NARA_KEY` → `naraBid.ts` 필드·낙찰조인 확정.
2. **실 sync 주기** — 쿼터 측정 후 결정(현재 API 502 회복 대기 자동 재시도 중). cron 트리거 배포 시 확정.
3. **phase 1.5** — 차고지 좌표(`config.GARAGE` 임시값) + 카카오 지오코딩(`distance_km`) + 지도.
4. **점수 계수** 실거래 보정.
5. **phase 2** — 리드 파이프라인(`linked_partner_id` 채움 → `partner`/`sale` 연결).
